## 2. CI 失败根因分析

### 2.1 时间线与症状

- 失败 run：id `27012743529`，job `79720099703`，commit `77dbf4c8999db7430b89e311f83353c456db73c1`
- 步骤时间：步骤 1 Set up job (3s) → 步骤 2 Checkout (1s) → 步骤 3 Setup Ruby (38s) → 步骤 4 Install bundler dependencies (1s) → **步骤 5 Build site (1s, failure)** → 步骤 6/7 skipped
- 关键观察：**Build site 步骤仅耗时 1 秒即失败**。在 105 篇文章 × 7 语言 × 11 layout 的规模下，正常构建至少需要 8-15 秒；1 秒失败只可能是 Jekyll **初始化阶段**（config 解析、plugin 加载、模板预编译）抛错。

### 2.2 嫌疑优先级（基于静态证据的 P0 故障树）

| 排序 | 嫌疑项 | 文件:行 | 证据 / 风险描述 | 验证方法 |
| --- | --- | --- | --- | --- |

| 🥉 #3 | `Gemfile.lock` 缺失 | `.gitignore:5` | 每次 `bundle install` 都重新解析依赖；新版本 jekyll / jekyll-sitemap / jekyll-seo-tag 引入 breaking change 时立即失败。**最常见的「突然 CI 失败」根因**。 | 删除 `.gitignore` 中的 `Gemfile.lock` 行，本地 `bundle install` 提交 `Gemfile.lock` |

| #5 | Liquid error 累积 | 多个 layout | `error_mode: warn` 应已抑制，但若某个 layout 在初始化阶段被 jekyll-include-cache 预编译时抛错，理论可能。 | 同上 |

### 2.3 根因结论（基于现有证据的合理推断）

**最可能组合**：`Gemfile.lock` 缺失（#3）+ `feed_content: excerpt` 触发 jekyll-feed 旧版本未识别（#1）。

**理由**：
1. `_config.yml:84-85` 的注释（中文）明确承认了 `feed_content: excerpt` 的兼容性问题：
   > 「jekyll-feed v0.17+ 支持 feed_content: excerpt。**GitHub Pages 默认带 jekyll-feed 全量版本，会输出全文**。如发现 feed.xml 仍是全文，可在每个 post front matter 中保留 excerpt 字段（已做），或在 _config.yml 升级 jekyll-feed 后改用 feed_content: excerpt。」
2. 这说明作者**当时就知道** GitHub Pages 默认带的 jekyll-feed 版本可能不识别 `feed_content: excerpt`。
3. 加上没有 `Gemfile.lock`，每次 CI 解析到的 jekyll-feed 版本都不同，可能某次解析到一个新版本（破坏性变化）或旧版本（不识别该键），导致构建失败。
4. 1 秒失败是 Jekyll 在解析 `feed:` 配置时立即抛 `NoMethodError: undefined method 'feed_content=' for ...` 的典型表现。

**建议的修复路径**（从最小到最大改动）：
1. **先** 删除 `_config.yml:89` 的 `feed_content: excerpt`（因为 front matter 里 `excerpt:` 已显式给出，feed 仍能正确截取）。
2. **然后** 取消 `.gitignore:5` 的 `Gemfile.lock`，让 CI 把 lock 文件 checkout 进来。
3. **最后** 评估是否还需要 `theme: minima`（若确认不需要则改为 `theme: null` 或删除该行）。
